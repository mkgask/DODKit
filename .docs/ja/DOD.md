# 決定指向開発

**Decision Oriented Development : DOD**

**持続可能な決定の蓄積を中心とした軽量な継続開発手法**

決定事項を継続開発の中心に据え、**議論フェーズ**と**実装フェーズ**を繰り返し、決定事項の一覧、決定に従属した実装、決定に至る経緯をそれぞれ維持することで、ソフトウェアの本当にあるべき姿を探し出す軽量開発手法です。

## 目的・背景・なぜDODか

DODの目的は、中長期的な開発においても軽量な決定一覧を維持し続けることで、次の決定に関しての認知負荷を極限まで下げ続けることです。

背景には、現在有効な決定事項、そこに至る経緯、実装詳細が同じ場所に混ざってしまうという、ソフトウェア開発でよくある失敗があります。これが起きると、「今なにが決まっているのか」「なぜそう決まったのか」という基本的な問いに答えるだけでも、毎回古い文脈を掘り返す必要が出ます。

DODはこの問題を、現在有効な決定事項と議論履歴を分離し、実装の正本を明確にすることで解決します。

## 中核原則

DODにおける本当の分岐点は、「その情報が重要かどうか」ではありません。重要なのは、その情報が次の実装判断を拘束するかどうかです。

DODでは、議論・決定・実装を分離して扱う必要があります。
- 議論は、調査、研究、質問、トレードオフ、理解の更新が蓄積する場です。
- 決定は、議論から生まれる拘束力のある定義です。
- 実装は、その決定をテストと動くコードに変換する行為です。

このためDODは、ドキュメントとして以下の2成果物を分離して作成します。

**決定事項リストファイル**
履歴を読まなくても次の実装判断を正しく行うために必要な、決定事項オブジェクトと拘束条件の集合です。

**議論履歴ファイル**
1つ以上の決定が生まれうる議論や調査の流れについて、理由、調査、トレードオフ、代替案、議論の履歴を残します。

1つの議論から、0個、1個、または複数の決定事項オブジェクトが発生しえます。

ある事実を議論履歴ファイルにだけ置いた結果、次の決定に際して議論履歴ファイルを読み返さなければならないなら、その事実は決定事項リストファイルで管理されるべきです。

## どうやってDODを実現するか

DODは次の3つの考え方で成り立ちます。
- 議論履歴と現在有効な決定事項を分離し、さらにその両方を実装作業から分離する
- 1つの議論から、独立して有効な複数の決定が生まれることを許容する
- 作業を議論フェーズと実装フェーズの2段階だけで進める

議論フェーズでは調査を行い、議論履歴を残しながら1つ以上の決定事項オブジェクトを更新します。実装フェーズでは、その決定を強制するテストとコードを書きます。現在有効な決定事項を明示し続けることで、人間もAIもすべての履歴を読み直さずに同じ正本から作業を始められます。

## 開発の流れ（2フェーズのみ）

**議論フェーズ**（現在の議論スコープに必要な決定が明示されるまで）
- 行動: 決定のための調査・研究・質問・議論（仕様だけでなく技術選定・制約なども対象）
- 成果物（必ずこの順で出力）
	1. **議論履歴ファイル（records/{議論ID}.md）** に経緯を記述（不変・追記のみOK/調査・研究などの際に新たな事実が判明するたびに記録OK）
	2. **決定事項リストファイル（DECISIONS.yml）** に現在有効な決定事項を更新し、議論の結果として独立して有効なルールが増えた場合は、1つ以上の新規決定またはサブ決定として追加する
- 制約: 関連する決定事項とその決定契約を DECISIONS.yml に明示するまで実装フェーズに入らない。議論履歴ファイルは経緯と補助文脈のために使う。

**実装フェーズ**（テストが通るまで）
- 行動: 設計、テストコード実装、コード実装
- 成果物
	- 決定を強制できるテストコード
	- 動くコード（全テストパス）
- 制約
	- 関連する決定から逸脱してはならない
	- DECISIONS.ymlおよび既存のテストコード、コードは十分尊重する
	- 新しい事実が判明したら、関連する議論履歴ファイルに追記し、拘束条件になったら DECISIONS.yml に昇格させる

## 決定契約

決定契約は現在有効な決定事項の一部なので、`DECISIONS.yml` に明示されていなければなりません。

議論履歴ファイルは経緯と補助文脈を保存するためのものであり、現在有効な実装拘束条件の正本ではありません。

1つの決定事項オブジェクトに収まらない場合は、契約を複数の決定事項オブジェクトまたはサブ決定に分解し、`records/{議論ID}.md` にだけ残す状態を避けます。

少なくとも、関連する決定事項オブジェクト群の中で次を明示します。
- 不変条件
- 非目標
- 受け入れ基準
- 失敗条件

## ドキュメント構造

DODは以下の2種類のドキュメントで実現します。

どちらに書くかは、次の基準で判定します。
- 次の実装判断に必要な情報なら `DECISIONS.yml` に置く
- 決定が生まれた議論、調査、トレードオフを説明する情報なら `records/{議論ID}.md` に置く
- 議論履歴側の情報が後から実装を拘束する条件になったら、決定事項オブジェクトまたはサブ決定として `DECISIONS.yml` に昇格させる

たとえばこうです。
- スコープ境界、非目標、不変条件、受け入れ基準、失敗条件は `DECISIONS.yml`
- 入出力仕様、対象環境、採用技術、配置先、命名規則、互換要件などの実装制約は `DECISIONS.yml`
- なぜその案を選んだか、どの代替案を捨てたか、どんな調査や議論があったかは `records/{議論ID}.md`
- 未解決論点、調査メモ、比較メモは、実装を拘束するまでは `records/{議論ID}.md` に置き、拘束条件になった時点で `DECISIONS.yml` に昇格させる

運用ルール:
1. この情報が `DECISIONS.yml` に無かったら、別の実装者は履歴を読まずに次の変更を正しく進められるか？
2. この情報が無かったら、テスト観点、非目標、制約、受け入れ挙動を取り違える可能性があるか？
3. この情報は過去の議論の説明ではなく、現時点で守るべき拘束条件か？

実装がぶれる可能性があるなら、安全側のデフォルトとして `DECISIONS.yml` に記載または昇格させます。

**決定事項リストファイル（DECISIONS.yml）**: 可変・プロジェクトの決定事項台帳
- このプロジェクトにおける決定事項オブジェクトの正本
- 履歴を読み直さなくても次の実装判断を正しく行うために必要な拘束条件の集合
- プロジェクトに関わる決定事項は、すべてここに決定事項オブジェクトまたはサブ決定として記述し、`status` で有効・保留・中止などの状態を表す
- 1つの議論や調査から生まれた複数の決定事項オブジェクトが、同じ議論履歴ファイルを参照してよい
- 新しい決定を作るときは、過去の経緯をすべて読み返さず、まずここだけを確認して始められる
- 実装時に履歴を読み直さなくて済むよう、決定契約もここに明示する
- トップレベルにカテゴリリスト
- 各カテゴリの中に決定事項オブジェクトの配列
- 決定事項オブジェクトの主なプロパティ
	- `id`: 必須 {category}-{連番}-{shortname} 決定IDと呼ぶ
	- `title`: 必須
	- `reason`: 必須 3行程度まで許容
	- `status`: 任意 基本は `⚠️Discussion In Progress` / `⚠️Discussion Approved` / `⚠️Implementing` / `✅️Implementation Approved` を使用し、`⛔️On Hold` や `⛔️Cancelled` は必要時のみ使用
	- `updated_at`: 任意
	- `link`: 任意 関連する議論履歴ファイルへのポインタ
- 決定事項オブジェクトは最大5層（推奨3層まで）ネスト可能
- 各決定事項オブジェクトは可能な限り簡潔に保つ
- 簡潔に保つとは件数を少なくすることではなく、1項目あたりを簡潔に保つことを意味する
- 少数の大きな決定より、小さな決定事項が多く並ぶことを優先する
- 実装に関わる決定事項は、すべて新規決定またはサブ決定としてDECISIONS.ymlに記載する
- 新たな拘束条件が見つかったら、議論履歴だけに埋めず、DECISIONS.ymlに追加する
- YAMLをテーブルビュー表示で人間も読みやすい

**議論履歴ファイル（records/{議論ID}.md）**: 不変・議論履歴
- 1つの議論または調査の流れにおける背景・調査・トレードオフ・議論の過程を記録する
- 一度書いたら基本的に追記のみで扱い、「当時なにが議論され、何が分かり、何が結論になったのか」を正確に振り返れる履歴として保つ
- 1つの議論履歴ファイルから、0個、1個、または複数の決定事項オブジェクトが DECISIONS.yml に生まれうる
- 自由記述形式（MADR風でもプレーンテキストでも可）
- 基本的に追記のみ。過去の事実として大切に残す
- 背景、調査、トレードオフ、代替案、なぜその決定が形成されたかの記録に比重を置く
- 実装を拘束する情報の唯一の置き場にせず、決定事項オブジェクト自体はDECISIONS.ymlに置く
- 決定契約がどのように形成されたかの補足は残してよいが、現在有効な契約の正本は DECISIONS.yml に置く

この分離により、**「このプロジェクトの決定事項は何か」と「なぜそうなったか」の両方を、必要なタイミングで最小のコストで得られる**ようになります。  
従来のSDDやADRではこれらが混在しており、新しい決定のための参考資料としては過去のノイズの取捨選択をする必要がありましたが、DODはこの問題を根本的に解決します。

## バージョン管理

**mainブランチ**
- DECISIONS.ymlと常に整合するようにする
- コードがテストを通り、関連決定が確定した時にだけマージする

**実装スコープごとの作業ブランチ**
- ブランチ名は実装スコープを表し、必要に応じて関連する議論IDや主要な決定IDを含める
- コードがテストを通り、関連決定が確定するまではこのなかだけで作業する

## 検証

検証はフェーズを作らずフックで強制する

**pre-commitフック**: テストとコード品質の検証
- 高レベル仕様のe2eテストとユニットテスト
- 型チェック、lint、コーディング規約などの細かいコードチェック

**pre-pushフック**: 決定の整合性を早期修正
- DECISIONS.ymlの整合性チェック
- DECISIONS.ymlと高レベル仕様テストの整合性チェック

## 注意事項

DODはADRではないです。
仕様駆動開発SDDでもないです。
意図駆動開発IDDでもないです。
これらは参考にしてもいいですが、あまり引っ張られないでください。
