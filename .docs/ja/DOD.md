# 決定指向開発

**Decision Oriented Development : DOD**

決定事項を開発の中心に据え、**議論フェーズ**と**実装フェーズ**をシンプルに繰り返すことで、目的に沿った真のソフトウェアを手に入れる軽量開発手法です。
決定を軽快にし、でも後から確実に振り返れる状態で管理し、AIと人間が一緒に素早く正確な開発を回します。

DODではドキュメントについて以下の2種類を記述します。

- **可変ファイル（DECISIONS.yml）**：現在有効な決定事項とその理由を一覧で管理  
  -> 新しい決定を作るときは**ここだけを見て即スタート**でき、過去の経緯を読み返す必要はありません。認知負荷が劇的に下がり、AIとも非常に相性が良いです。

- **不変ファイル（records/{決定ID}.md）**：決定に至った背景・調査・トレードオフ・議論の過程を記録  
  -> 一度書いたら基本的に追記のみ。後から「当時はなぜこう決めたのか」を正確に振り返れる歴史として残します。

この分離により、**「今何が有効か」と「なぜそうしたか」の両方だけを、必要なタイミングで最小のコストで得られる**ようになります。  
従来のSDDやADRではこれらが混在しており、新しい決定のための参考資料としては過去のノイズの取捨選択をする必要がありましたが、DODはこの問題を根本的に解決します。

## 開発の流れ（2フェーズのみ）

**議論フェーズ**（決定を確定するまで）
- 行動：決定のための調査・研究・質問・議論（仕様だけでなく技術選定・制約なども対象）
- 成果物（必ずこの順で出力）：
  1. **records/{決定ID}.md** に経緯を記述（不変・追記のみOK/調査・研究などの際に新たな事実が判明するたびに記録OK）
  2. **DECISIONS.yml** に決定事項と理由を更新（可変）
  3. 決定を実現するためのテストコードを先に作成（主にe2eテスト、ユニットテストも書けるところは書く）
- 制約：テストコードの作成が完了するまでコードは書かない
- 制約：必要以上の不要な深堀りはしない

**実装フェーズ**（テストが通るまで）
- 行動：テストコードを通すためのコード実装（リファクタ含む）、テストコード追加が必要ならどんどん追加してOK
- 成果物：動くコード（全テストパス）
- 制約：DECISIONS.md、既存のテストコードは十分尊重する、records/{決定ID}.mdは追記のみOK

## ドキュメント構造

**DECISIONS.yml**（可変・現在有効一覧）
- トップレベルにカテゴリリスト
- 各カテゴリの中に決定事項オブジェクトの配列
- オブジェクトの主なプロパティ：
  - `id`（{category}-{連番}-{shortname} 決定IDと呼ぶ）, `title`, `decision`, `reason`（3行程度まで許容）, `status`（Accepted / Superseded など）, `updated_at`, `link`
- YAMLをテーブルビュー表示で人間も読みやすい

**records/{決定ID}.md**（不変・経緯記録）
- 自由記述形式（MADR風でもプレーンテキストでも可）
- 基本的に追記のみ。過去の事実として大切に残す

## バージョン管理

**mainブランチ**
- DECISIONS.ymlと常に整合するようにする
- コードがテストを通り、決定が確定した時にだけマージする

**個別作業ブランチ**
- ブランチ名は `{決定ID}`
- コードがテストを通り、決定が確定するまではこのなかだけで作業する
