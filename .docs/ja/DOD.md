# 決定指向開発

**Decision Oriented Development : DOD**

決定事項を開発の中心に据え、**議論フェーズ**と**実装フェーズ**をシンプルに繰り返すことで、目的に沿った真のソフトウェアを手に入れる軽量開発手法です。
決定を軽快にし、でも後から確実に振り返れる状態で管理し、AIと人間が一緒に素早く正確な開発を回します。

DODではドキュメントについて以下の2種類を記述します。

**決定事項リストファイル（DECISIONS.yml）**: 現在有効な決定事項とその理由を一覧で管理  
可変です。新しい決定を作るときは**ここだけを見て即スタート**でき、過去の経緯を読み返す必要はありません。認知負荷が劇的に下がり、AIとも非常に相性が良いです。

**決定履歴ファイル（records/{決定ID}.md）**: 決定に至った背景・調査・トレードオフ・議論の過程を記録  
不変です。一度書いたら基本的に追記のみ。後から「当時はなぜこう決めたのか」を正確に振り返れる歴史として残します。

この分離により、**「今何が有効か」と「なぜそうしたか」の両方だけを、必要なタイミングで最小のコストで得られる**ようになります。  
従来のSDDやADRではこれらが混在しており、新しい決定のための参考資料としては過去のノイズの取捨選択をする必要がありましたが、DODはこの問題を根本的に解決します。

## 開発の流れ（2フェーズのみ）

**議論フェーズ**（決定を確定するまで）
- 行動: 決定のための調査・研究・質問・議論（仕様だけでなく技術選定・制約なども対象）
- 成果物（必ずこの順で出力）
	1. **決定履歴ファイル（records/{決定ID}.md）** に経緯を記述（不変・追記のみOK/調査・研究などの際に新たな事実が判明するたびに記録OK）
	2. **決定事項リストファイル（DECISIONS.yml）** に現在有効な決定事項を更新し、議論の結果として独立して有効なルールが増えた場合は新規決定またはサブ決定として追加する
- 制約: 決定契約（不変条件、非目標、受け入れ基準、失敗条件）を確定し決定履歴ファイルに記述し、DECISIONS.ymlに決定事項を記載するまで実装フェーズに入らない、コードを書き始めない

**実装フェーズ**（テストが通るまで）
- 行動: 設計、テストコード実装、コード実装
- 成果物
	- 決定を強制できるテストコード
	- 動くコード（全テストパス）
- 制約
	- 関連する決定から逸脱してはならない
	- DECISIONS.ymlおよび既存のテストコード、コードは十分尊重する
	- 新しい事実が判明したら、決定履歴ファイルに追記する

## ドキュメント構造

**決定事項リストファイル（DECISIONS.yml）**: 可変・現在有効一覧
- トップレベルにカテゴリリスト
- 各カテゴリの中に決定事項オブジェクトの配列
- 決定事項オブジェクトの主なプロパティ
	- `id`: 必須 {category}-{連番}-{shortname} 決定IDと呼ぶ
	- `title`: 必須
	- `reason`: 必須 3行程度まで許容
	- `status`: 任意 基本は `⚠️Discussion In Progress` / `⚠️Discussion Approved` / `⚠️Implementing` / `✅️Implementation Approved` を使用し、`⛔️On Hold` や `⛔️Cancelled` は必要時のみ使用
	- `updated_at`: 任意
	- `link`: 任意 関連する決定履歴ファイルへのポインタ
- 決定事項オブジェクトは最大5層（推奨3層まで）ネスト可能
- 各決定事項オブジェクトは可能な限り薄く保つ
- 薄く保つとは件数を少なくすることではなく、1項目あたりを簡潔に保つことを意味する
- 少数の大きな決定より、小さな決定事項が多く並ぶことを優先する
- 新たに現在有効なルールが見つかったら、決定履歴だけに埋めず、新規決定またはサブ決定としてDECISIONS.ymlに追加する
- YAMLをテーブルビュー表示で人間も読みやすい

**決定履歴ファイル（records/{決定ID}.md）**: 不変・経緯記録
- 自由記述形式（MADR風でもプレーンテキストでも可）
- 基本的に追記のみ。過去の事実として大切に残す
- 背景、調査、トレードオフ、代替案、なぜその決定が形成されたかの記録に比重を置く
- 現在有効なサブ決定の唯一の置き場にせず、現在有効なルールはDECISIONS.ymlに置く
- 決定契約は決定履歴ファイルに残す
	- 不変条件: 絶対守ること
	- 非目標: やらないこと
	- 受け入れ基準: 観測可能な目標値
	- 失敗条件: 不合格範囲を確定

## バージョン管理

**mainブランチ**
- DECISIONS.ymlと常に整合するようにする
- コードがテストを通り、決定が確定した時にだけマージする

**個別作業ブランチ**
- ブランチ名は `{決定ID}`
- コードがテストを通り、決定が確定するまではこのなかだけで作業する

## 検証

検証はフェーズを作らずフックで強制する

**pre-commitフック**: テストとコード品質の検証
- 高レベル仕様のe2eテストとユニットテスト
- 型チェック、lint、コーディング規約などの細かいコードチェック

**pre-pushフック**: 決定の整合性を早期修正
- DECISIONS.ymlの整合性チェック
- DECISIONS.ymlと高レベル仕様テストの整合性チェック

## 注意事項

DODはADRではないです。
仕様駆動開発SDDでもないです。
意図駆動開発IDDでもないです。
これらは参考にしてもいいですが、あまり引っ張られないでください。
