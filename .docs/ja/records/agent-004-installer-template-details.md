# Decision Record: agent-004-installer-template-details

## メタデータ
- Status: Discussion Approved
- Date: 2026-04-04
- Scope: インストーラーテンプレート資産の構成と出力マッピング

## 背景とリサーチ
インストーラー入力としてテンプレートファイルを使う決定自体はすでに有効ですが、テンプレート詳細はまだ十分に定義されていません。
テンプレート配置が暗黙のままだと、インストーラー実装が場当たり的なファイル処理へ流れやすくなります。

この決定は `agent-002-installer-delivery` から分離します。理由は、テンプレート詳細が単なる履歴メモではなく、独立して現在有効になるルールだからです。

現時点で分かっている制約:
- インストーラーの初期対象CLIは GitHub Copilot Chat のみ。
- インストーラー出力はワークスペースのカスタマイズパスへ対応付ける必要がある。
- 将来のCLI拡張を見込むため、テンプレート構造は拡張可能であるべき。

## 決定
インストーラー実装前に、テンプレートファイル詳細を独立決定として定義します。

この決定で定義する対象:
- テンプレートファイルをリポジトリ内のどこに置くか
- テンプレートファイルをインストール先パスへどう対応付けるか
- CLI別バリアントをどう表現するか
- どのファイルをそのままコピーし、どのファイルをレンダリングまたはパラメータ化するか

## 決定契約
### 不変条件
- テンプレートのソースファイルとインストール後の出力ファイルは分離する。
- 出力マッピングは決定的で、追跡可能であること。
- 初期構成は GitHub Copilot Chat を支えつつ、将来のCLI追加を妨げないこと。

### 非目標
- この決定で複数CLI対応の最終設計までは確定しない。
- この決定でインストーラー実行ロジックまでは実装しない。
- 具体的な第2CLIがない段階で、過剰なテンプレート抽象化を入れない。

### 受け入れ基準
- リポジトリ内のテンプレート配置場所が定義される。
- 各出力ファイルに、対応するソーステンプレートまたは生成ルールがある。
- GitHub Copilot Chat 向けテンプレートを曖昧さなく識別できる。
- 将来のCLI拡張パスが構造上見える。

### 失敗条件
- テンプレートソースと出力ファイルが混在している。
- ファイル対応付けが暗黙ルールや属人的知識に依存する。
- 構造が1CLI固定を前提にしており、後から拡張できない。

## オープンクエスチョン
- テンプレートファイルはトップレベルの `templates/` に置くべきか、よりインストーラー寄りのパスに置くべきか。
- どのファイルをリテラルコピーし、どのファイルにプレースホルダー置換を許可するべきか。
- CLIターゲット名をパスやメタデータ上でどう表現するべきか。

## 議論解決（2026-04-04）
現在スコープに対する解決結果:
- 初期テンプレートパスは `templates/agent.md` を採用する。
- 初期リリースでコピー対象が1ファイルでも、コピー対象リストはインストーラー内に保持する。
- コピー元は `raw.githubusercontent.com` を使用する。
- コピー先パスは、インストーラーが実行されたカレントディレクトリからの相対パスとして解決する。

実装上の解釈:
- 初期構成では、過剰なディレクトリ分割を入れず最小構成を維持する。
- コピー対象が増えたときにも対応関係を明示できるよう、インストーラー内のマニフェスト的リストを必須とする。
- コピー元解決とコピー先解決は別責務として扱い、片方から暗黙推論しない。

現在の決定への反映:
- 上記オープンクエスチョンは、初期リリース範囲では解決済みとする。
- この解決から生じる現在有効なルールは、`DECISIONS.yml` 上で `agent-004-installer-template-details` 配下のサブ決定として管理する。

このリポジトリにおけるコピー元URLの確認例:
- `https://raw.githubusercontent.com/mkgask/DODKit/refs/heads/main/templates/agent.md`
- `https://raw.githubusercontent.com/mkgask/DODKit/main/templates/agent.md`

上記2形式はいずれも検証で HTTP 200 を返した。再現性を重視する場合は、コミット固定の `<ref>` を推奨する。
